# Source Posts And Messages Decision Packet R1

Status: Active decision packet
Owner: Founder + product engineering
Last updated: 2026-06-24

## Purpose

Clarify whether artists, bands, promoters, and other managed source entities can
post messages to followers in the current MVP.

This packet prevents agents from confusing:

- long-term source doctrine, where sources can issue follower updates;
- current source-dashboard runtime, which is intentionally smaller; and
- socket/chat prototypes, which are not the source follower-update product.

## Decision

Source posts/messages are product-valid later, but not current MVP runtime.

Current MVP launch source tools remain:

- `Release Deck`
- `Source Profile`
- `Print Shop`
- `Registrar`

Do not add a source posts/messages composer, source inbox, follower message
thread, community chat, notification system, or broadcast-message endpoint to the
current source-dashboard launch slice.

## Current Runtime Truth

Current source-facing runtime:

- Source Dashboard is the current operating shell for managed source entities.
- Release Deck creates source-owned tracks through the current URL-only MVP seam.
- Print Shop creates source-owned or promoter-lane events through explicit
  source/promoter eligibility.
- Registrar remains reachable from source-side context.
- Artist Profile exposes source identity, direct listening, follow/share, links,
  events, and capped music rows.

Current runtime does not provide:

- source-authored follower updates;
- source posts on Artist Profile;
- source posts on Plot Feed;
- follower-only message delivery;
- source notification inbox/outbox;
- persistent source-message models;
- source-specific socket rooms or moderation workflow.

## How To Read Existing Source Doctrine

`docs/solutions/MVP_SOURCE_AND_FEED_RULES_FOUNDER_LOCK_R1.md` preserves the
long-term source model:

- sources may have follower updates;
- first source entry can create a community-wide update through the community
  layer;
- after first entry, source-origin announcements should go to followers only;
- follower updates may contain freeform source messaging.

That doctrine is still useful, but it does not activate current runtime.

For current MVP implementation, the active Source Dashboard contract wins:

- no fake analytics, billing, growth, or social-message modules;
- no source-management tools inside the listener profile;
- no source posts/messages feature until this packet is superseded by an
  implementation spec.

## Boundary Rules

If source posts/messages are activated later:

- They are source-authored updates to followers, not listener-authored community
  chat.
- They are follower-only after the initial source-entry announcement.
- They must not become community-wide feed noise by default.
- They must not become `Blast`, `Recommend`, `Collect`, `Play It Loud`, or
  `Upvote`.
- They must not affect Fair Play, rotation, tier progression, registrar
  authority, or community governance.
- They must preserve source ownership and operator permission checks.
- They must be separable from future business promotions, paid offers, coupons,
  and analytics products.

## Explicit Non-Goals For Current MVP

- Do not add a Source Dashboard posts card.
- Do not add an Artist Profile post wall.
- Do not add a Plot Feed composer for artists/promoters.
- Do not activate socket `community-message` or direct-message prototypes as
  source follower updates.
- Do not add notification persistence or follower-delivery fanout.
- Do not implement business follower updates as a shortcut to business runtime.
- Do not create placeholder CTAs such as `Coming Soon`, `Post Update`, or
  `Message Followers` in current runtime.

## Future Activation Checklist

Before implementing source posts/messages, create or update a dedicated spec
covering:

1. Source eligibility
   - Which source types can post.
   - Which operator roles can create, edit, hide, or delete posts.

2. Audience and routing
   - Follower-only delivery after initial source entry.
   - Whether source posts also appear on Artist Profile, follower inbox, or a
     later source-update feed.
   - How first-entry community-wide announcements are generated by the
     community/system layer.

3. Data model
   - Persistent post/update model.
   - Source ownership and operator audit fields.
   - Visibility, moderation, edit/delete, and tombstone behavior.

4. Delivery
   - Whether delivery is pull-based, notification-based, socket-backed, email,
     or a combination.
   - Backpressure and fanout behavior for large follower counts.

5. Interaction grammar
   - Which reactions, if any, are allowed.
   - Explicit exclusion from signal actions and Fair Play.

6. Tests
   - Source operator permission tests.
   - Follower-only visibility tests.
   - No listener-profile/source-tool leakage tests.
   - No Plot Feed/community-wide leakage tests unless explicitly scoped.

## Agent Guidance

For current source-dashboard work:

- Load `docs/agent-briefs/ARTIST_PROFILE_SOURCE_DASHBOARD.md`.
- Treat posts/messages as deferred.
- Do not infer implementation permission from socket chat prototypes or old
  source doctrine.
- Preserve the current live tool card set unless this packet is superseded by an
  approved implementation spec.

For future source-posts work:

- Use this packet as the starting decision record.
- Load the source/feed founder lock for doctrine.
- Add a real spec before runtime implementation.
