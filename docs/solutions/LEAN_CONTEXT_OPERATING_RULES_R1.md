# Lean Context Operating Rules

Status: Active
Owner: Founder + product engineering
Last updated: 2026-04-07

## Purpose
Keep Codex sessions usable by reducing avoidable context growth and moving durable truth into the repo early.

These rules are operational. They do not replace canon, specs, founder locks, or handoff discipline.

## Rules
1. Do not paste full transcripts unless necessary
- Paste only the decisive fragment.
- Summarize the rest in 1-3 lines.

2. Salvage early
- Once a decision is real, write it to:
  - founder lock
  - handoff
  - changelog
- Stop carrying it in chat after that.

3. One thread, one job
- Doctrine thread = decisions.
- Implementation thread = code.
- Do not keep both mixed longer than necessary.

4. Trust written locks
- Once a lock is written to repo docs, reference it.
- Do not re-argue it from memory every few turns.

5. Keep tool output minimal
- Read only the lines needed.
- Avoid full-file dumps and giant status outputs unless debugging truly needs them.

6. Summarize before reset
- Before compaction or thread switch, capture:
  - locks
  - open decisions
  - next actions
  - drift corrections

7. Escalate only real ambiguity
- Do not burn context on low-risk clarification loops.
- Make the reasonable default and note it.

8. Use fresh sessions intentionally
- After a dense doctrine phase, start a clean implementation session.
- Do not drag the whole ontology thread into code work.

9. Separate durable truth from exploratory thought
- Only durable truth belongs in docs.
- Exploration stays short and disposable.

10. Treat giant MCP or plugin inventories as background noise unless proven otherwise
- Do not keep debating them in-thread unless you are actively reconfiguring Codex.
